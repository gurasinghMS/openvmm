## 1. Table Overview

| Property | Value |
|---|---|
| **Table name** | `HyperVWorkerTable` |
| **Cluster** | `azcore.centralus.kusto.windows.net` |
| **Database** | `Fa` |
| **Full path** | `cluster('azcore.centralus.kusto.windows.net').database('Fa').HyperVWorkerTable` |
| **Process that logs** | `vmwp.exe` — Hyper-V's **worker process** that hosts the runtime management of a Virtual Machine |
| **Provider(s) collected** | Defined in `MdsFa.xml`; primary provider is `Microsoft.Windows.HyperV.Worker` (also `Microsoft-Windows-Hyper-V-Worker`) |
| **Content** | All events and traces generated by vmwp.exe, **except** EventId 23100–23145 (virtual-device-specific data duplicated from VdevOperation EventSource / TraceLogging) |
| **Payload format** | Event payload is presented in **JSON format** through the `Message` column |

### What it contains
HyperVWorkerTable captures the full lifecycle of VMs from the worker-process perspective:
- VM start / stop / reset events
- Guest crash and panic reports
- UEFI watchdog timeout notifications
- Underhill (VTL2) start errors and crash events
- Underhill servicing status (success, failure, reset)
- Live migration state transfer events
- VM-to-ID mapping
- Worker process stages and operations

## 2. Key Fields

| Field | Description | Usage |
|---|---|---|
| `PreciseTimeStamp` | High-precision event timestamp | Primary time filter; use `between (start .. end)` |
| `TIMESTAMP` | Event timestamp (slightly lower precision) | Alternative time column; some queries use this |
| `NodeId` | GUID of the physical host node | **Always filter on this** to scope to a specific host |
| `Message` | JSON-formatted event payload | Contains ContainerId/VmId inside JSON; use `has` or `contains` to filter |
| `EventMessage` | Human-readable event description | Sometimes empty; use `iif(isnotempty(EventMessage), EventMessage, Message)` |
| `EventId` | Numeric event identifier | Key for filtering specific event types (see §3) |
| `Level` | Event severity level (e.g., 4=Info, 3=Warning, 2=Error) | Filter for errors with `Level <= 2` |
| `TaskName` | Name of the task category | e.g., `VmNameToIdMapping`, `VmControl`, `Migrate Virtual Machine` |
| `Opcode` | Operation code (1=Start, 2=Stop/End for task stages) | Track stage begin/end in servicing flows |
| `Pid` | Worker process ID | Critical for scoping to a specific VM's worker; a node runs one vmwp per VM |
| `Tid` | Thread ID | Further scoping within a worker process |
| `ProviderName` | ETW provider name | e.g., `Microsoft.Windows.HyperV.Worker`, `Microsoft-Windows-Hyper-V-Worker` |
| `ChannelName` | ETW channel name | Used in detailed queries |
| `ActivityId` | Correlation activity GUID | Correlate related events across tables |
| `RelatedActivityId` | Parent/related activity GUID | Trace causal chains |

### Tips on field usage
- **`Message` vs `EventMessage`**: The `Message` field always has content (JSON blob). `EventMessage` is a pre-formatted description that may be empty. Best practice is to project both: `EventMessage = iif(isnotempty(EventMessage), EventMessage, Message)`.
- **Filtering by ContainerId/VmId**: These IDs are embedded inside the JSON `Message` field. Use `Message has fn_containerId` or `Message has fn_vmId` to filter. Do **not** expect a dedicated `ContainerId` column.
- **Getting the worker PID for a container**: Use a subquery to resolve PID from containerId, then filter all worker logs by that PID (see §4).

## 3. All Event IDs

### VM Lifecycle Events

| EventId | Symbolic Name | Description | Source Page |
|---|---|---|---|
| **18500** | MSVM_VM_STARTED | `'<containerId>' started successfully. (Virtual machine ID <vmId>)` — VM started | Running Container Unhealthy |
| **18504** | (Shutdown IC) | `'<containerId>' was shut down using the Shutdown Integration Component with parameters: Force = true, Reason = '...'` — VM clean shutdown via IC | Stop Container Failure |
| **18514** | MSVM_GUEST_RESET_SUCCESS | `'<containerId>' was reset by the guest operating system. (Virtual machine ID <vmId>)` — Guest OS initiated a reset | Running Container Unhealthy, Migration Failure |

### Guest Crash / Error Events

| EventId | Symbolic Name | Description | Source Page |
|---|---|---|---|
| **18539** | MSVM_TRIPLE_FAULT_GENERAL_ERROR | General triple-fault error in the guest | Running Container Unhealthy |
| **18540** | MSVM_TRIPLE_FAULT_UNSUPPORTED_FEATURE_ERROR | Triple fault due to unsupported feature | Running Container Unhealthy |
| **18550** | MSVM_TRIPLE_FAULT_INVALID_VP_REGISTER_ERROR | Triple fault due to invalid VP register | Running Container Unhealthy |
| **18560** | MSVM_TRIPLE_FAULT_UNRECOVERABLE_EXCEPTION_ERROR | Unrecoverable exception triple fault | Running Container Unhealthy |
| **18590** | MSVM_GUEST_CRASH_REPORT | Guest crash/panic report. `Message` JSON contains `VmErrorMessage` with the guest's last logs before crash (e.g., Underhill panic traces). For Underhill VMs, the Linux kernel invokes `/bin/underhill-crash` which writes a dump via VMBus Crash Dump Device to the host. | Underhill TSG, Underhill Servicing |
| **18602** | MSVM_GUEST_CRASH_DUMP_SUCCESS | Guest crash dump was generated successfully | Running Container Unhealthy |
| **18604** | MSVM_GUEST_CRASH_DUMP_FAILURE | Guest crash dump failed | Running Container Unhealthy |

### UEFI / Watchdog Events

| EventId | Symbolic Name | Description | Source Page |
|---|---|---|---|
| **18600** | (Watchdog Timeout) | `'<containerId>' has encountered a watchdog timeout and was reset.` — UEFI boot-services 2-minute watchdog fired. Provider: `Microsoft-Windows-Hyper-V-Chipset`. Also appears in HyperVWorkerTable for Gen2/Legacy HCL VMs. | UEFI Watchdog, Running Container Unhealthy |
| **18601** | (Boot event) | Related boot event | Running Container Unhealthy |
| **18606** | (INT19 transition) | INT19 transition to boot loader attempted (Gen1 VMs) | Running Container Unhealthy |
| **18610** | (HCL fatal error) | `fatal virtual firmware error` — Host Compatibility Layer fatal error. Provider: `Microsoft-Windows-Hyper-V-Worker`. Indicates HCL fault in Trusted Launch / Confidential VMs. | Running Container Unhealthy |
| **18617** | (Liveness watchdog) | Same text as 18600 but different event ID. Introduced in AH2023 for workload liveness checks — this is a **guest OS issue**, not UEFI. Route via RDOS Route. | Running Container Unhealthy |

### Underhill (VTL2) Events

| EventId | Symbolic Name | Description | Source Page |
|---|---|---|---|
| **18620** | MSVM_START_VTL0_REQUEST_ERROR | VM's VTL0 failed to start. `Message` JSON contains `ErrorCode`, `Status`, `ResultDocument` (may contain Underhill error like `Underhill.StorageCannotOpenVtl2Device`). | Underhill TSG |

### Underhill Servicing Events

| EventId | Symbolic Name | Description | Source Page |
|---|---|---|---|
| **5124** | (Servicing failure) | Failed Underhill servicing operation. `EventMessage` shows: stage at which failure occurred, Underhill current/new version, correlation ID, error code. Example: `failed to reload management VTL at stage: SaveManagementVtlState with error code 0x800704C7` | Underhill Servicing |
| **5126** | (Servicing success) | Successful Underhill servicing. `EventMessage` contains correlation ID (use to query Underhill logs). | Underhill Servicing |
| **5128** | (VM reset after servicing failure) | Evidence that VM was reset due to Underhill servicing failure. Occurs after event 5124. | Underhill Servicing |
| **5136** | (Guest power event during servicing) | Underhill servicing failed due to guest-initiated power event. `EventMessage` shows reason (e.g., `Reason: TripleFaultReset`). | Underhill Servicing |

### Live Migration Events

| EventId | Symbolic Name | Description | Source Page |
|---|---|---|---|
| **21324** | (LM state transfer start) | `Started transferring virtual machine state for VM '<vmId>'.` | Migration Failure |
| **21325** | (LM state transfer end) | `Finished transferring virtual machine state for VM '<vmId>' (<size> KByte), time <seconds> s.` — Use to correlate transfer size vs. duration for performance analysis. | Migration Failure |

## 4. All Kusto Query Examples

### 4.1 Basic HyperVWorkerTable Query (from Hyper-V Kusto Queries page)

The canonical starting point for any investigation:

```kql
let fn_faultTime = datetime(5/6/2023 12:12:49 PM);
let fn_delta = 3h;
let fn_startTime = fn_faultTime - fn_delta;
let fn_endTime = fn_faultTime + fn_delta;
let fn_nodeId = "9a9c671e-f9d5-4f4f-9eb4-8e968618124e";
let fn_containerId = "5954a243-632f-4604-9f7f-5be55b5e6685";
let fn_vmId = "d679e7e3-787a-46bb-badf-e21989a9568c";
cluster('azcore.centralus.kusto.windows.net').database("Fa").HyperVWorkerTable
| where PreciseTimeStamp between ((fn_startTime) .. (fn_endTime))
| where NodeId == fn_nodeId
| where Message has fn_containerId or Message has fn_vmId
| project PreciseTimeStamp, TaskName, EventId, Message, Opcode
```

### 4.2 Check if Guest Reset the VM (EventId 18514) — Running Container Unhealthy

```kql
// Check if the reset was done by guest OS itself?
// Presence of event id 18514 indicates that there was a guest reset
// (VM was reset by the guest OS itself).
let fn_faultTime = datetime(2024-05-01 1:00:00.000);
let fn_startTime = fn_faultTime - 1h;
let fn_endTime = fn_faultTime + 1h;
let fn_nodeId = "ac13a952-9a4a-918c-437f-8ffb55a192aa";
let fn_containerId = "fa0f4c12-c406-41aa-8eae-275f2f5a638e";
cluster('azcore.centralus').database("Fa").HyperVWorkerTable
| where PreciseTimeStamp between (fn_startTime .. fn_endTime) and EventId == "18514"
| where NodeId == fn_nodeId
| where EventMessage has fn_containerId
| project PreciseTimeStamp, EventId, EventMessage
| sort by PreciseTimeStamp asc
```

### 4.3 Stop Container — Worker Process Logs

Get the worker PID first, then query all worker logs:

```kql
let fn_nodeId = "e266c292-20f8-4c4c-bdfb-d30a32934050";
let fn_containerId = "a154a5df-dcd7-45fa-b7b9-4d164425165d";
let fn_faultTime = datetime(2023-12-08T19:20:14.470016Z);
let fn_startTime = fn_faultTime - 1h;
let fn_endTime = fn_faultTime + 1h;
cluster('azcore.centralus').database('Fa').HyperVWorkerTable
| where NodeId == fn_nodeId and PreciseTimeStamp between (fn_startTime .. fn_endTime)
//| where Message has fn_containerId
| where Pid == <Insert worker process PID here>
| project PreciseTimeStamp, Pid, Tid, ProviderName, EventId, TaskName, Level, Opcode,
    EventMessage = iif(isnotempty(EventMessage), EventMessage, Message)
```

#### Subquery to get the worker PID for a container:

```kql
cluster('azcore.centralus').database('Fa').HyperVWorkerTable
| where (PreciseTimeStamp > fn_startTime) and (PreciseTimeStamp < fn_endTime)
    and NodeId == fn_nodeId
| where Message has fn_containerId
| project Pid;
```

### 4.4 Stop Container — VMMS + Worker Combined with PID Resolution

```kql
let fn_nodeId = "77764c5d-a46e-2207-4857-e5cc64c7b154";
let fn_containerId = "05b62c9a-64f9-466e-89fc-c59586c7cf71";
let fn_faultTime = datetime(2023-03-31T02:42:00Z);
let fn_startTime = fn_faultTime - 1.3h;
let fn_endTime = fn_faultTime + 2h;
let pid = toscalar(
    cluster('azcore.centralus').database('Fa').HyperVWorkerTable
    | where (PreciseTimeStamp > fn_startTime) and (PreciseTimeStamp < fn_endTime)
        and NodeId == fn_nodeId
    | where Message has fn_containerId
    | project Pid
);
cluster('azcore.centralus').database('Fa').HyperVWorkerTable
| where (PreciseTimeStamp > fn_startTime) and (PreciseTimeStamp < fn_endTime)
    and NodeId == fn_nodeId
| where Pid == pid
| project TIMESTAMP, Level, TaskName, Opcode, Message, EventMessage
| sort by TIMESTAMP asc
```

### 4.5 Guest Error Events — Running Container Unhealthy

```kql
let fn_faultTime = datetime(2022-10-07 02:09:00.0000000);
let fn_startTime = fn_faultTime - 1h;
let fn_endTime = fn_faultTime + 1h;
let fn_nodeId = "3532d7cf-ae39-4a1d-bf6d-7b76f0970fd2";
let fn_containerId = "47e03d2b-2eae-4322-bd68-0a88b627af4a";
let fn_vmId = "1C940AD7-76A8-46C0-AA71-A7827B075C81";
let fn_guestErrors = dynamic([
    18540, // MSVM_TRIPLE_FAULT_UNSUPPORTED_FEATURE_ERROR
    18550, // MSVM_TRIPLE_FAULT_INVALID_VP_REGISTER_ERROR
    18539, // MSVM_TRIPLE_FAULT_GENERAL_ERROR
    18560, // MSVM_TRIPLE_FAULT_UNRECOVERABLE_EXCEPTION_ERROR
    18590, // MSVM_GUEST_CRASH_REPORT
    18602, // MSVM_GUEST_CRASH_DUMP_SUCCESS
    18604, // MSVM_GUEST_CRASH_DUMP_FAILURE
    18514  // MSVM_GUEST_RESET_SUCCESS
]);
cluster('azcore.centralus').database("Fa").WindowsEventTable()
| where NodeId == fn_nodeId
| where PreciseTimeStamp between (fn_startTime .. fn_endTime)
    and ProviderName == "Microsoft-Windows-Hyper-V-Worker"
    and EventId in (fn_guestErrors)
| where (Description has fn_containerId or Description has fn_vmId)
| project PreciseTimeStamp, NodeId, Description, Level, EventId
```

> **Note:** This query uses `WindowsEventTable` (not HyperVWorkerTable) but filters for `Microsoft-Windows-Hyper-V-Worker` provider. The same EventIds appear in HyperVWorkerTable.

### 4.6 UEFI Watchdog Timeout — Find Watchdog Event (Gen2 / Legacy HCL)

```kql
let fn_nodeId = "c077f606-c595-f110-58d5-6eb250a384f1";
let fn_containerId = "b5bf2ec6-327c-4dac-a9d2-b6210e9e487e";
let fn_vmId = "7f751308-e91d-45f9-b21b-926a83eeb94f";
let fn_faultTime = datetime(2025-06-27 09:06:43.182686700+00:00);
let fn_startTime = fn_faultTime;
let fn_endTime = fn_faultTime + 14d;
cluster('azcore.centralus').database('Fa').HyperVWorkerTable
| where NodeId in (fn_nodeId)
| where Message has_any(fn_containerId) or Message has_any(fn_vmId)
| where PreciseTimeStamp between (fn_startTime .. fn_endTime)
| project PreciseTimeStamp, Pid, Tid, ProviderName, EventId, TaskName, Level, Opcode,
    EventMessage = iif(isnotempty(EventMessage), EventMessage, Message)
| where EventMessage has_any("has encountered a watchdog timeout")
| order by PreciseTimeStamp asc
```

```kql

```

### 4.8 Underhill VTL0 Start Error (EventId 18620) — Underhill TSG

```kql
let fn_startTime = datetime(09-29-2023 07:35);
let fn_endTime = datetime(09-30-2023 21:35);
let fn_nodeId = "452f9e06-865b-b10e-5983-b61050b8e495";
let fn_containerId = "6714F5D1-4CDB-4302-9733-5F6C9A2BCEE5";
cluster('azcore.centralus').database('Fa').HyperVWorkerTable
| where TIMESTAMP between (fn_startTime .. fn_endTime)
| where NodeId == fn_nodeId
| where EventMessage has fn_containerId
| where EventId == 18620 //MSVM_START_VTL0_REQUEST_ERROR
| extend MessageParsed = parse_json(Message)
| project
    TIMESTAMP,
    ProviderName,
    Level,
    EventId,
    ErrorCode = MessageParsed.ErrorCode,
    Status = MessageParsed.Status,
    ResultDocument = MessageParsed.ResultDocument,
    Message,
    EventMessage
```

The `ResultDocument` subfield may contain the Underhill error, e.g.:
```json
{
  "error_id": "Underhill.StorageCannotOpenVtl2Device",
  "message": "cannot open nvme namespace: ...",
  "file_name": "underhill/underhill_core/src/dispatch/vtl2_settings_worker.rs",
  "line": 675
}
```

### 4.12 Underhill Servicing — Check VmControl/ReloadManagementVtl Task

Verify the servicing call reached the worker process:

```kql
let fn_startTime = datetime(2024-05-14T07:15:06.4888086Z);
let fn_endTime = datetime(2024-05-14T07:17:06.4888086Z);
let fn_vmId = "158b311b-f347-4ffe-92f3-9b335e7c76d5";
cluster('azcore.centralus').database('Fa').HyperVVmmsTable
| union cluster('azcore.centralus').database('Fa').HyperVWorkerTable
| where TIMESTAMP between (fn_startTime .. fn_endTime)
| where Message has fn_vmId
| where TaskName == "VmControl" and Message contains "ReloadManagementVtl"
| project TIMESTAMP, Message, TaskName
```

### 4.13 Underhill Servicing — Worker Stages with Correlation ID

```kql
let fn_impactTime = datetime("2024-05-07T16:35:19.5889444Z");
let fn_startTime = fn_impactTime - 10m;
let fn_endTime = fn_impactTime + 10m;
let fn_nodeId = "3294bd07-9f08-501d-d62c-c2bcea0cb027";
let fn_vmId = "cdc43451-f401-4a64-9795-ef5b2eb239a1";
let fn_correlationId = "44a45e3e-2d47-4419-833d-9c4e644955f8";
cluster('azcore.centralus').database('Fa').HyperVWorkerTable
| where TIMESTAMP between (fn_startTime .. fn_endTime)
| where NodeId == fn_nodeId
| where Message has fn_vmId and Message has fn_correlationId
| project TIMESTAMP, NodeId, TaskName, Opcode
```

> **Opcode 1** = stage start, **Opcode 2** = stage end. If Opcode 2 is missing, the stage is stuck or exceeded the PilotFish timeout (60s).

### 4.15 Live Migration — Common LM Query (VMMS + Worker + VID)

```kql
let fn_faultTime = datetime(2023-07-20T13:31:06.7758456Z);
let fn_startTime = fn_faultTime - 1s;
let fn_endTime = fn_faultTime + 1s;
let fn_nodeIdSrc = "02786dd2-1dd9-89ee-57e5-7638b3736b51";
let fn_containerIdSrc = "80d8cd73-db9f-40d7-bb1d-8b0783004fd7";
let fn_nodeIdDest = "c8f5e105-d49d-d145-999b-3de509250e30";
let fn_containerIdDest = "ae81451e-3abc-4767-ad01-86cf156911dd";
let fn_vmId = "583b7d5e-6ff9-4744-b665-3e9c71ea4bd9";
union
    (cluster('azcore.centralus').database('Fa').HyperVVmmsTable
     | where PreciseTimeStamp between (fn_startTime .. fn_endTime)
         and NodeId in (fn_nodeIdSrc, fn_nodeIdDest)
         and Message has_any(fn_containerIdSrc, fn_containerIdDest, fn_vmId)),
    (cluster('azcore.centralus').database('Fa').HyperVWorkerTable
     | where PreciseTimeStamp between (fn_startTime .. fn_endTime)
         and NodeId in (fn_nodeIdSrc, fn_nodeIdDest)),
    (cluster('azcore.centralus').database('Fa').HyperVVidTable
     | where PreciseTimeStamp between (fn_startTime .. fn_endTime)
         and NodeId in (fn_nodeIdSrc, fn_nodeIdDest))
| extend LMNode = iif(NodeId == fn_nodeIdSrc, "Source", "Destination")
| project PreciseTimeStamp, Pid, Tid, LMNode, ProviderName, EventId, TaskName,
    Level, Opcode,
    EventMessage = iif(isnotempty(EventMessage), EventMessage, Message)
| order by PreciseTimeStamp asc
```

## 5. Filtering and Using the Table Effectively

### Essential filter pattern
Every query should include at minimum:
```kql
HyperVWorkerTable
| where PreciseTimeStamp between (fn_startTime .. fn_endTime)
| where NodeId == fn_nodeId
| where Message has fn_containerId or Message has fn_vmId
```

### Recommended projections
```kql
| project PreciseTimeStamp, Pid, Tid, ProviderName, EventId, TaskName, Level, Opcode,
    EventMessage = iif(isnotempty(EventMessage), EventMessage, Message)
```

### Filter by specific event type
```kql
| where EventId == 18590  // guest crash
| where EventId in (5124, 5126, 5128, 5136)  // underhill servicing
| where EventMessage has_any("has encountered a watchdog timeout")  // watchdog
```

### Parse JSON Message payload
```kql
| extend MessageParsed = parse_json(Message)
| project ErrorCode = MessageParsed.ErrorCode, Status = MessageParsed.Status
```

## 6. Common Investigation Patterns

### Pattern 1: "Did the VM start?"
1. Query `MycroftContainerHealthSnapshot` for container state transitions
2. Query `SharedWorkspace.AgentOperations()` for agent perspective
3. Query `SharedWorkspace.HyperVContainerStarted()` for Hyper-V's perspective (looks for EventId 18500)
4. If VM failed to start → follow Start Container Failure TSG

### Pattern 2: "Why did the VM reboot unexpectedly?"
1. Check for **EventId 18514** (guest reset) in HyperVWorkerTable
2. Check timing relative to "container started" message:
   - If reset is **>5 minutes** after start → likely guest OS issue
   - If reset is **soon after** start → check VM generation, proceed to HCL/UEFI checks
3. Check for **EventId 18600** (watchdog timeout) → UEFI boot issue
4. Check for guest errors (18539, 18540, 18550, 18560, 18590, 18602, 18604)

### Pattern 3: "Why did stop container fail?"
1. Query `IfxOperationV2v1EtwTable` for agent operations around fault time
2. Query `HyperVVmmsTable` for VMMS-level shutdown task events
3. Query HyperVWorkerTable for **EventId 18504** (shutdown via IC)
4. Get worker PID via subquery → filter all worker logs by PID
5. Check time between shutdown start and completion

### Pattern 4: "Underhill VM crash"
1. Check **EventId 18620** (VTL0 start error) — look at `ResultDocument`
2. Check **EventId 18590** (guest crash) — look at `VmErrorMessage` in JSON
3. Cross-reference with `UnderhillEventTable` for VTL2 logs
4. Check Watson dumps for crash dump analysis

### Pattern 5: "Underhill servicing failure"
1. Query **EventId 5124** (failure) / **5126** (success) / **5128** (VM reset) / **5136** (guest power event)
2. Extract correlation ID from EventMessage
3. Verify servicing reached worker: look for `TaskName == "VmControl"` with `ReloadManagementVtl`
4. Use correlation ID to trace through `UnderhillEventTable`
5. Check Opcode 1/2 pairs for stuck stages (missing Opcode 2 = stuck or timed out)

### Pattern 6: "Live migration failure"
1. Get ContainerIds/NodeIds from `LiveMigrationSessionCompleteLog`
2. Get VmId via `VmNameToIdMapping` task in HyperVWorkerTable
3. Run Common LM Query (Worker + VMMS + VID) on both source and destination
4. Look for errors in worker logs, especially storage/vdev save/restore failures
5. Check EventId 18514 for guest reset during migration (aborts LM)
6. Analyze EventId 21325 for state transfer performance

### Pattern 7: "UEFI watchdog timeout"
1. Find watchdog event (`has encountered a watchdog timeout`) in HyperVWorkerTable
2. Look back **2–2.5 minutes** before the timestamp for UEFI activity
3. For OpenHCL VMs, also check `UnderhillEventTable`
4. Check boot order, NIC/storage errors, device configuration

## 8. Tips and Gotchas

### Data retention and availability
- Hyper-V logs in Kusto age out over time. If a VM has been running for >60 days, start logs may not be available.
- `HyperVCounters` table is **currently not being populated** due to a pipeline issue — you may not see results.

### EventId exclusions
- EventId **23100–23145** are **excluded** from HyperVWorkerTable (virtual device specific data; see VdevOperation EventSource).

### ContainerId vs VmId
- **ContainerId** = Agent's ID for a VM incarnation on a node (multiple containers can map to the same customer VM).
- **VmId** = Hyper-V internal GUID, local to the node and temporal. There can be **multiple VMIDs** for a given container over time.
- Hyper-V logs generally use VmId. Determine VmId before investigation using the `VmNameToIdMapping` query (§4.14).

### Message field JSON
- Always parse `Message` as JSON when you need structured access: `parse_json(Message)`.
- The `Message` field for crash events (18590) contains a `VmErrorMessage` subfield with guest kernel logs.
- The `Message` field for VTL0 errors (18620) contains `ResultDocument` with Underhill error details.

### Opcode semantics in servicing
- **Opcode 1** = stage start, **Opcode 2** = stage end.
- A **missing Opcode 2** for a given stage means the stage is **stuck or exceeded the PilotFish timeout (60 seconds)**.

### Event 18617 vs 18600
- **18600** = Genuine UEFI boot watchdog timeout (Hyper-V issue or UEFI/config issue).
- **18617** = Introduced in AH2023; same text but for workload liveness — this is a **guest OS issue**, not UEFI.

### Filtering considerations for Live Migration
- In LM, **not all logs contain VmId or ContainerId**. You may need to filter by NodeId + time window only.
- Large VMs (TB-scale) can have brownout up to **9 hours**, so adjust your time window accordingly.
- Use `Message !has "frutilities" and Message !has "frctutilities"` to reduce noise.

### Confidential VMs
- For SEV-SNP and TDX confidential VMs, **crash dumps are NOT sent to the host**. Watson dumps will not be available.

### Worker PID
- Each VM has its own `vmwp.exe` process. When investigating a specific VM, **resolve the PID first** using the subquery pattern, then filter all worker logs by that PID to reduce cross-VM noise.

### SharedWorkspace functions
- The `azcore.centralus / SharedWorkspace` database contains convenience functions maintained by the Hyper-V SME v-team:
  - `HyperVContainerStarted(nodeId, containerId, startTime, endTime)` — checks if VM started
  - `AgentOperations(nodeId, containerId, startTime, endTime)` — agent's view of operations
  - `HyperVEvents(nodeId, containerId, startTime, endTime)` — aggregated Hyper-V logs

### Escalation paths
- **Hyper-V SME team:** `hypsme`
- **HCL issues:** Hyper-V MVM team (`hypmvm`)
- **Guest OS (Windows):** `WSD CFE\HCCompute-Guest OS Health`
- **Guest OS (Linux):** `LSG/Triage`
- **Agent issues:** `OneFleet Node/AzureHost-VMService-Sev-3-4` (or Sev-1-2)
- **Confidential VMs:** `OneFleet Node/ConfidentialComputing-Sev-3-4`
- **General routing:** See `aka.ms/rdosroute`
